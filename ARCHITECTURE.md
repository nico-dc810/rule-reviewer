# 产品架构草图

## 目标

这个应用不是简单做“关键词匹配”，而是要解决一个更难的问题：

把用户口中的模糊要求，例如：

- “更专业”
- “更像老板要的方案”
- “这段代码要更稳一点”
- “文章要更可信，也要更容易执行”

转换成一套可解释、可执行、可复核的审查规则，然后再判断输入内容是否达标。

## 推荐的四层结构

### 1. 需求理解层

输入：

- 模糊需求
- 内容类型
- 使用场景
- 目标受众

输出：

- 归一化需求
- 关键维度
- 约束条件
- 不确定点

这一层的目标是把“模糊话术”变成结构化意图。

建议输出格式：

```json
{
  "goal": "让方案更专业且更可执行",
  "audience": "业务负责人",
  "dimensions": ["clarity", "completeness", "actionable", "credibility"],
  "constraints": ["不能太长", "要能直接开会使用"],
  "unknowns": ["是否需要行业数据支持"]
}
```

### 2. 规则生成层

输入：

- 结构化需求
- 内容类型

输出：

- 审查规则列表
- 每条规则的权重
- 评分方式
- 失败定义

建议规则对象：

```json
{
  "id": "R1",
  "name": "actionable",
  "description": "内容必须包含可执行动作和下一步建议",
  "weight": 0.25,
  "check_method": "llm_rubric",
  "pass_criteria": "至少给出 3 个明确动作",
  "examples": ["先做渠道验证，再扩量", "列出本周执行步骤"]
}
```

### 3. 审查执行层

这一层不要只靠一种方法。
推荐使用“混合审查引擎”：

- LLM 审查：判断抽象质量，比如是否专业、是否有说服力
- 规则引擎：执行确定性检查，比如是否有标题、是否包含结论、是否包含风险提示
- 专用分析器：
  - 代码：AST、lint、test、复杂度、依赖风险
  - 文章：结构分析、事实核验、引用检查
  - 方案：章节覆盖、行动项抽取、指标识别

输出：

- 分项得分
- 总分
- 证据
- 缺陷
- 修改建议

### 4. 人机协同层

这是最终产品很重要的一层。

因为“模糊需求”天然有歧义，所以系统应该允许：

- 用户查看系统生成的规则
- 用户手动修改权重
- 用户删除不合理规则
- 用户补充“这次特别在意什么”
- 用户确认最终审查标准

这会显著提高可信度和可控性。

## 适合的产品流程

1. 用户输入模糊需求
2. 系统生成结构化理解结果
3. 系统生成审查规则草案
4. 用户确认或编辑规则
5. 用户提交待审查内容
6. 系统输出审查报告
7. 用户选择“自动重写”或“继续人工修改”

## MVP 和正式版的区别

### 当前 MVP

- 通过关键词和启发式方法推断规则
- 用简单打分逻辑给出结果
- 重点是验证交互链路是否成立

### 正式版建议

- 接入大模型生成结构化 rubric
- 针对不同内容类型走不同审查器
- 保存历史版本和审查记录
- 支持“同一要求批量审查多个生成物”
- 支持“按组织标准沉淀规则模板”

## 一个很重要的产品判断

你真正做的，不只是“审查工具”。

更准确地说，它是：

**一个把模糊主观要求转成可执行评估标准的审查系统。**

这个定位会比“AI 审稿器”或者“AI 代码审查”更有延展性，因为它天然可以覆盖：

- 代码
- 文章
- 方案
- 需求文档
- Prompt 输出
- 运营文案

## 下一步建议

如果你要继续做，我建议下一步优先补这 3 个能力：

1. 规则确认界面
2. LLM 版规则生成器
3. 内容类型专用审查器接口

这样这个项目就会从“演示原型”变成“真正有产品势能的底座”。
